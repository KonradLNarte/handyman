  # Research Task: The Evolution of Specifications — From Voice Recording to Buildable Blueprints

  You are a research analyst studying how natural language specifications evolve through
  iterative human-AI collaboration. Your subject is the Resonansia project — a real ERP
  system built entirely from specs fed to AI coding agents.

  ## Your Mission

  Trace the COMPLETE arc of specification evolution across this project — from a raw voice
  conversation between two people brainstorming at a table, all the way to phase specs that
  an AI agent executes to produce working code. Analyze what changed at each transformation,
  what was gained, what was lost, and what universal principles we can extract about writing
  specs that compile cleanly into working code on the first pass.

  The ultimate question: **How do we go from a vague conversation to a buildable spec in
  as few lossy steps as possible?**

  ## Research Questions

  ### 1. THE FULL TRANSFORMATION CHAIN
  Trace what happened at each step and analyze the nature of each transformation:
  - Gen 0 → Gen 1: **Conversation → Product spec.** How did a messy, tangential voice
    conversation become a structured product spec? What did Claude extract? What did it
    miss? What implicit knowledge in the conversation (tone, emphasis, Kimmo's hesitations,
    the back-and-forth negotiation of scope) was lost in transcription and structuring?
  - Gen 1 → Gen 2: **Product spec → Technical architecture.** How did a tech-agnostic
    "what/why" document become a specific technical architecture with 7 tables, event
    sourcing, and bitemporality? Was this transformation faithful to the original vision?
    Did technical decisions distort the product intent?
  - Gen 2 → Gen 3: **Architecture → Bootstrap instructions.** How did a system spec become
    an executable bootstrap instruction? What structural patterns emerged (ARTIFACT/VERIFY)?
  - Gen 3 → Gen 4: **System spec → Phase specs.** How did one Claude instance decompose
    the master spec into buildable phases? Did later phases learn from earlier ones?
    Compare Phase 1, 2, 3, and 4 specs directly — what conventions were added, what
    mistakes were repeated?

  ### 2. THE INFORMATION LOSS CASCADE
  Each transformation is a lossy compression. Trace specific pieces of information:
  - Find something Kimmo said in the voice recording that made it all the way to Phase 3
    code. What's the longest surviving thread?
  - Find something Kimmo emphasized in the conversation that was LOST and never made it
    into any spec. What's the most significant loss?
  - Does the Swedish → English transition lose domain nuance? (Kimmo's industry jargon,
    Swedish construction terms, cultural context like BRF/ROT/RUT)
  - Is there implicit business logic in the conversation (things Kimmo assumes everyone
    knows) that no spec ever captured?

  ### 3. THE FORMALITY GRADIENT
  Each generation sits at a different point on the formality spectrum:
  - Gen 0: Raw spoken Swedish, stream of consciousness, interruptions, tangents
  - Gen 1: Structured Swedish, RFC 2119 modals, requirement IDs, but still "what/why"
  - Gen 2: English, AXIOM/INTERFACE/SCHEMA/RULE blocks, technical architecture
  - Gen 3: ARTIFACT/INVARIANT/VERIFY/GIT execution pattern
  - Gen 4: Same pattern with increasing specificity (type signatures, function names)

  Map this gradient. Is there an optimal formality level for each transformation?
  Where does over-specification become counterproductive? Is there a "formality cliff"
  where going one step further in precision yields diminishing returns?

  ### 4. THE "CONVERSATION TO SPEC" PIPELINE
  The pipeline today is: conversation → product spec → tech spec → bootstrap → phases.
  That's 4 transformations, each with information loss and potential for drift.
  - Could the pipeline be shortened? Could a single AI instance go from the voice
    recording directly to buildable phase specs?
  - What would be lost by shortening the pipeline?
  - What would be GAINED? (Less drift, fewer contradictions, faster iteration)
  - What are the irreducible human judgment points that cannot be automated?
  - Imagine you had to build a "spec factory" that takes a 1-hour conversation with a
    domain expert and produces phase specs ready for an AI coding agent. Design that
    pipeline. What are the stages? What's automated vs human-reviewed?

  ### 5. THE DOMAIN EXPERT'S VOICE
  Kimmo is the domain expert. He knows painting, customers, BRF politics, ROT deductions,
  seasonal cycles, pricing, field logistics.
  - How much of Kimmo's expertise survives into the final specs?
  - Are there moments in the conversation where Kimmo explains something critical that
    the spec writer (Claude) would never have known to ask about?
  - What questions SHOULD have been asked in that conversation but weren't?
  - If you were designing a "domain expert interview protocol" to maximize spec quality,
    what would the interview template look like?

  ### 6. THE PATTERN LANGUAGE
  Across all generations, identify the recurring structural patterns:
  - Which patterns appear in Gen 0 (the conversation!) and survive to Gen 4? (durable)
  - Which patterns were invented at each generation? (emergent)
  - Which patterns were tried and abandoned? (failed)
  - Can you define a minimal "spec grammar" that captures the essential patterns?
  - How do the .claude/skills/ files relate to the spec documents? Are they a separate
    pattern or a compression of existing patterns?

  ### 7. TOWARD A SPEC FACTORY
  The ultimate goal: a system where a domain expert describes what they want (possibly
  just by talking), and a "spec factory" produces buildable phase specs that an AI coding
  agent can execute with minimal friction.
  - Based on everything you've read, what would this factory's pipeline look like?
  - What would the "intermediate representation" be? (The Gen 2 tech spec? Something new?)
  - How do you handle the bootstrapping problem? (The first spec has no codebase to
    reference; later specs must reference existing code)
  - What role does the voice recording play? Is it a viable input format for a spec
    factory, or does it need to be a structured interview?
  - How close is this project to achieving that vision?

  ## Output Format

  Write a research report with:
  1. **Executive Summary** (1 page) — The 5 most important insights about spec evolution
  2. **The Transformation Chain** (detailed analysis of each Gen N → Gen N+1 transition,
     with specific quotes from the voice recording traced through to phase specs)
  3. **Information Loss Audit** (what survived, what was lost, what was never captured)
  4. **Pattern Catalog** (table: pattern name, first appeared in, survived to, effectiveness)
  5. **The Formality Curve** (visual/textual model of optimal formality by spec type)
  6. **The Domain Expert Interview Protocol** (template for future domain expert sessions)
  7. **The Spec Factory Blueprint** (proposed pipeline from conversation to buildable spec)
  8. **Recommendations** — Concrete, ranked changes to the spec writing process
