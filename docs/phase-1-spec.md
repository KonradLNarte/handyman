# Resonansia — Phase 1: Foundation

> Implementation spec for the core data layer, auth, and minimal UI.
> After this phase: you can create tenants, add nodes and edges,
> register events, correct events, and see project economics on a
> working dashboard. All behind RLS with tenant isolation verified.

## 2. Database Client & Utilities

### Artifact

```
ARTIFACT db_client:
  File: packages/db/client.ts

  Exports two database access patterns:
  
  1. `db` — Drizzle ORM instance connected via DATABASE_URL.
     Used in Server Components and Server Actions.
  
  2. `createSupabaseServerClient(cookieStore)` — Supabase client
     that reads the user's auth cookie. Used for RLS-aware queries.
  
  3. `createSupabaseServiceClient()` — Supabase client with
     service_role key. Used ONLY for:
     - Tenant creation (before user has a tenant_id in JWT)
     - Migration scripts
     - Background jobs
     NEVER in user-facing API handlers or Server Actions.

ARTIFACT uuid_util:
  File: packages/shared/utils/uuid.ts
  
  Exports: generateId() -> string
  Uses import { v7 as uuidv7 } from 'uuid' to generate UUIDv7 IDs.
  All IDs in the system MUST be generated by this function.
  The database MUST NOT generate IDs (no gen_random_uuid() defaults).

ARTIFACT db_helpers:
  File: packages/db/queries/helpers.ts
  
  Exports:
  - withTenant(tenantId: string) — returns Drizzle where clause for tenant_id
  - computeTotal(qty: number | null, unitPrice: number | null) -> number | null
    Deterministic: returns qty * unitPrice, or null if either is null.
    This is the ONLY function that computes event totals.
```

### Invariants

```
INVARIANT service_role_restricted:
  createSupabaseServiceClient() MUST NEVER be called from:
  - Server Actions
  - React Server Components that serve user requests
  - Route Handlers that serve user requests
  It is for background/admin operations only.

INVARIANT ids_from_app:
  All UUIDs MUST be generated in the application layer via generateId().
  Database columns MUST NOT have DEFAULT gen_random_uuid().
  This ensures UUIDv7 with embedded transaction time.

INVARIANT total_from_helper:
  Event total MUST always be computed by computeTotal().
  Never inline qty * unitPrice anywhere else.
```

## 3. Authentication

### Artifact

```
ARTIFACT auth_flow:
  Supabase Auth with email/password. Social login is SHOULD (skip for Phase 1).
  
  Files:
    apps/web/app/(auth)/login/page.tsx      — Login page (Server Component + Client form)
    apps/web/app/(auth)/register/page.tsx    — Registration page
    apps/web/app/(auth)/layout.tsx           — Centered auth layout
    apps/web/lib/auth.ts                     — Auth helper functions
    apps/web/middleware.ts                    — Protect routes, redirect unauthenticated

  Registration flow:
  1. User enters email + password
  2. Supabase creates auth.users entry
  3. A database trigger (or Server Action post-registration) creates:
     - A new Tenant (status=active)
     - A new Node (type=org) as the tenant's organization
     - A new Node (type=person) for the user
     - An Edge (type=member_of, role=owner) linking person → org
  4. Custom JWT claim is set: tenant_id = the new tenant's id
     (Via Supabase's auth.hook or custom claims in app_metadata)
  
  Login flow:
  1. User enters email + password
  2. Supabase authenticates
  3. JWT includes tenant_id in claims
  4. Middleware reads JWT, redirects to /dashboard if authenticated

ARTIFACT auth_tenant_claim:
  The tenant_id MUST be embedded in the Supabase JWT.
  
  Implementation options (choose one that works with local Supabase):
  A. Supabase custom claims via auth.users.raw_app_meta_data
  B. A Postgres function that injects tenant_id into JWT claims
  
  The claim MUST be readable by RLS policies via:
    current_setting('request.jwt.claims', true)::json->>'tenant_id'
  
  This is what makes all RLS policies work. Without it, every query returns empty.

ARTIFACT auth_middleware:
  File: apps/web/middleware.ts
  
  - Checks for valid Supabase session
  - Redirects unauthenticated users to /login
  - Allows /login and /register without auth
  - Refreshes session tokens on each request
```

### Invariants

```
INVARIANT jwt_has_tenant:
  After login, the JWT MUST contain tenant_id.
  Every RLS policy depends on this. If it's missing, the app is broken.

INVARIANT registration_atomic:
  Tenant + org node + person node + owner edge MUST be created
  in a single transaction. If any step fails, all roll back.

INVARIANT auth_pages_minimal:
  Auth pages are functional, not beautiful. Phase 7 handles polish.
  Email + password form, basic validation, error messages. That's it.
```

### Verify

```
VERIFY auth:

  2. Navigate to /dashboard without auth → redirected to /login
  3. Register a new user → tenant + org + person + edge created in database
  4. Login with new user → JWT contains tenant_id claim
  5. After login → redirected to /dashboard (can be empty page for now)
  6. Second user registers → different tenant_id, cannot see first user's data
```

## 7. Event System

This is the most critical section. Events are the economic source of truth.

### Artifact

```
ARTIFACT event_creation:
  File: packages/core/events/create.ts
  
  createEvent(input: CreateEventInput) -> Event
    - Generates id via generateId() (UUIDv7 = transaction time)
    - Sets tenant_id from JWT
    - Validates type_id is valid event_type label
    - Validates data against type-specific Zod schema
    - Computes total = computeTotal(qty, unitPrice)
    - Sets origin (human, ai_generated, system, external_api)
    - Validates occurred_at is a valid timestamp
    - If ref_id is set: validates it points to a ROOT event
      (an event with ref_id = null in the same tenant)
    - Inserts into events table
    - Returns created event

ARTIFACT event_correction:
  File: packages/core/events/correct.ts
  
  correctEvent(originalEventId: string, newValues: CorrectionInput) -> Event
    - Looks up the original event
    - If original has ref_id: follows it to the root (ensures root pointer)
    - Creates a new event with:
      - type_id = 'adjustment' label
      - ref_id = root event id (NOT the original if original was itself an adjustment)
      - New qty/unit_price/total values
      - data.reason (required — why the correction was made)
      - origin = 'human'
      - occurred_at = same as root event's occurred_at (correcting the same moment)
    - Returns the correction event

ARTIFACT active_event_resolution:
  File: packages/core/events/resolve.ts
  
  Uses the getActiveEvents query from packages/db/queries/active-events.ts
  (created during bootstrap).
  
  Exports:
  - getActiveEventsForNode(tenantId, nodeId, typeIds?) -> ActiveEvent[]
  - getActiveEventsForProject(tenantId, projectId) -> ActiveEvent[]
    (includes events on the project AND all child nodes)

ARTIFACT project_economics:
  File: packages/core/economics/calculate.ts
  
  calculateProjectEconomics(tenantId: string, projectId: string) -> ProjectEconomics
  
  Returns:
    quotedTotal:    number  — SUM of active quote_line events
    timeCost:       number  — SUM of active time events (total)
    materialCost:   number  — SUM of active material events (total)
    invoicedTotal:  number  — SUM of active invoice_line events
    margin:         number  — quotedTotal - (timeCost + materialCost)
    marginPercent:  number  — margin / quotedTotal * 100 (or 0 if no quote)
  
  ALL sums use getActiveEventsForProject (resolving corrections).
  ALL arithmetic in TypeScript — never delegated to AI.

ARTIFACT event_types_dispatch:
  File: packages/shared/schemas/event-data-dispatch.ts
  
  getEventDataSchema(typeCode: string) -> ZodSchema
    'time'          → eventDataTimeSchema
    'material'      → eventDataMaterialSchema
    'photo'         → eventDataPhotoSchema
    'message'       → eventDataMessageSchema
    'quote_line'    → eventDataQuoteLineSchema
    'invoice_line'  → eventDataInvoiceLineSchema
    'adjustment'    → eventDataAdjustmentSchema
    'state_change'  → eventDataStateChangeSchema
    'payment'       → eventDataPaymentSchema
    'note'          → eventDataNoteSchema
    default         → z.record(z.unknown())
```

### Invariants

```
INVARIANT event_append_only:
  The events table MUST NOT have UPDATE or DELETE operations anywhere
  in the codebase. grep -r "UPDATE.*events" and "DELETE.*events"
  MUST return zero results (except in GDPR erasure code, Phase 5).

INVARIANT event_total_deterministic:
  total is ALWAYS computed by computeTotal() in the application layer.
  It MUST equal qty * unit_price. It MUST NOT be accepted from user input.

INVARIANT event_root_pointer:
  correctEvent MUST always set ref_id to the ROOT event.
  If the user tries to correct an adjustment, follow ref_id to the root first.

INVARIANT event_origin_honest:
  Events created by user actions: origin = 'human'
  Events created by AI after approval: origin = 'ai_generated'
  Events created by system automation: origin = 'system'
  Events from external integrations: origin = 'external_api'
  These MUST be set correctly. They are not changeable after creation.

INVARIANT economics_from_active_only:
  calculateProjectEconomics MUST use only active events
  (after correction chain resolution). Shadowed events
  MUST NOT contribute to sums.
```

### Verify

```
VERIFY events:
  1. Create a time event (qty=8, unit_price=650) → total=5200, origin='human'
  2. Create a time event with wrong total → rejected (total recomputed, or ignored)
  3. Create an adjustment (ref_id=event1, qty=6) → new event with ref_id=event1
  4. getActiveEventsForNode returns only the adjustment (qty=6), not original (qty=8)
  5. Create second adjustment (ref_id=event1, qty=7) → active value is 7
  6. calculateProjectEconomics returns correct sums using only active events
  7. Attempt UPDATE on events table → fails (verify via RLS or application check)
  8. Create event in tenant A → not visible from tenant B
  
VERIFY correction_chain:
  Scenario from spec (S-13):
  1. Create event A: type=time, qty=8
  2. Create event B: type=adjustment, ref_id=A, qty=6
  3. Create event C: type=adjustment, ref_id=A, qty=7
  4. Verify: B.ref_id = A.id AND C.ref_id = A.id (both point to root)
  5. Verify: active value = 7 (C has highest UUIDv7)
  6. Verify: project economics use qty=7, not 8 or 6 or 21
```
